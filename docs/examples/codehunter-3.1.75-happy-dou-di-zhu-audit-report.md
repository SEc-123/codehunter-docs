## Executive Summary

This report presents the security audit findings for **Happy Dou Di Zhu**, a browser-based multiplayer Chinese Landlord poker game built with PHP 5.4+, MySQL (MyISAM), jQuery, and cookie-based session management.

**Total findings reviewed:** 30

| Confirmation Status | Count |
| --- | --- |
| Confirmed | 18 |
| Highly Likely | 6 |
| Needs More Evidence | 4 |
| Rejected / Excluded | 2 |

**Key risks identified:**

- **Supply chain and code deployment** (RC-001-REV-001): The auto-update mechanism fetches remote XML and downloads/extracts ZIP patches with TLS verification disabled, no integrity checks, and no URL allowlisting, enabling full server compromise via MITM or update server compromise.
- **SQL injection** (RC-001-REV-004, RC-001-REV-005, RC-001-REV-006): Multiple SQL injection vectors confirmed in the installer, application db helper, and admin SQL execution panel, all using string concatenation without prepared statements.
- **Authentication and session management** (RC-001-REV-008, RC-001-REV-009, RC-001-REV-022, RC-001-REV-023): Unsalted MD5 password hashing, missing cookie security flags, and cookie-derived CSRF tokens create an exploitable session management chain.
- **Race conditions in financial flows** (RC-001-REV-025): MyISAM storage engine with read-then-update patterns and no concurrency controls enables double-spend via concurrent seat claims and balance deductions.

## Appendix

### A. Application Overview

**Happy Dou Di Zhu** is a monolithic PHP web application implementing a browser-based multiplayer Chinese Landlord poker game. The application uses a custom MVC-like architecture with PATH_INFO routing, MySQL database storage (MyISAM engine), cookie-based session management, and AJAX polling for real-time game state updates.

**Technology Stack:**

- **Server:** PHP 5.4+ with custom routing via index.php
- **Database:** MySQL via PDO (MyISAM engine)
- **Client:** jQuery, Layer.js modal library, HTML/CSS/JavaScript
- **Session Management:** Cookie-salt based session tokens stored in MySQL session table
- **External Dependencies:** cURL for update checks, SMTP for password recovery emails

### B. Scope of Review

The audit covered all server-side PHP source code in the application, including:

- 14 business functions across player-facing, game logic, admin, and system maintenance domains
- 20 subfunctions covering authentication, game rooms, card gameplay, currency management, admin operations, and system installation
- Database schema (9 tables: account, session, shield, stat, member, buy, bill, talk, room)
- Client-side JavaScript for game interaction and polling
- ZIP extraction library for update patching
- SMTP mail library for password recovery

### C. Confirmation Standard

Each finding was evaluated against four confirmation tiers:

1. **Confirmed:** Complete source-to-sink evidence chain with control gap verified in source code.
2. **Highly Likely:** Strong structural evidence present but exploitability depends on runtime conditions (filesystem permissions, browser behavior, server configuration) not observable in source review.
3. **Needs More Evidence:** Preliminary indicators exist but insufficient evidence to confirm or refute the finding.
4. **Rejected / Excluded:** Evidence refutes the candidate or context is insufficient and the candidate was deprioritized.
