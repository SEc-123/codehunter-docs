## Executive Summary

This security audit analyzed the Vexus cryptocurrency exchange platform — a Java 8-based multi-module application covering spot trading, perpetual contracts, C2C trading, wallet management, withdrawal, blockchain integration, and DApp operations. The platform deploys via WAR files on Apache Tomcat with Dubbo RPC and ZooKeeper for inter-module communication.

**37 findings** were evaluated across the full codebase:

- **26 Confirmed** findings with direct source-level evidence
- **5 Highly Likely** findings with strong supporting evidence but limited by source visibility constraints
- **6 Needs More Evidence** findings with partial evidence requiring deeper runtime or source verification
- **0 Rejected** findings

The confirmed findings reveal **4 critical-severity vulnerabilities** (OS command injection in database backup, arbitrary file read via path traversal, wallet double-spending via Redis race condition, and withdrawal double-spend), alongside multiple high-severity issues spanning authentication weaknesses (MD5 hashing, token exposure, missing CSRF), authorization bypass (default-allow model), and systemic concurrency defects across all financial subsystems.

The most pervasive class of vulnerability is **concurrency and idempotency risk**: the platform uses Redis-first write patterns with async database synchronization but no distributed locking, creating TOCTOU (time-of-check-to-time-of-use) race conditions across wallet, exchange, contract, C2C, and withdrawal operations. In-memory JVM-local locks (ExchangeLock, C2cLock, ContractLock) provide insufficient protection, especially in multi-instance deployments.

### RC-001-REV-001: OS Command Injection via Database Backup Shell Command Construction

| Field | Detail |
|---|---|
| **ID** | RC-001-REV-001 |
| **Title** | OS Command Injection via Database Backup Shell Command Construction |
| **Severity** | Critical |
| **Bound Function** | Database Backup (BF-010) |
| **Entry Point** | Configuration parameter: `db.username`, `db.password` from `db.properties` enter the backup shell command construction pipeline |
| **Sensitive Point / Risk Sink** | `backupDBDataByShell` in `MysqlBackupImpl.java` (lines 190–238): assembled shell command string passed to `CMDUtil.runCMD` for `Runtime.exec` execution |
| **Control Missing / Failure** | No shell escaping, no parameterized command construction, no input sanitization for shell metacharacters; single-quote wrapping of password can be bypassed |
| **Evidence & Data Flow** | Source: `db.properties` lines 6–7 contain `db.username=root`, `db.password=2uVS6UkYyV7Do4JbN9zebDUMoF4w88q3CNxanZ0wEnY%3D`. Transit: `parserMapParams` (lines 170–190) constructs params map from db.properties values without sanitization. Sink: `backupDBDataByShell` (lines 190–238) concatenates params into `StringBuffer` shell command, then calls `CMDUtil.runCMD(cmdBuf.toString())`. |
| **Why Confirmed** | Config values are directly concatenated into shell commands without escaping, enabling command injection via config file modification. |
| **Business Impact** | An attacker who can modify `db.properties` (via file write vulnerability or deployment pipeline compromise) can execute arbitrary OS commands on the host during backup operations, gaining full system control. |
| **Remediation** | Replace shell command construction with `ProcessBuilder` using argument arrays. Implement allowlist validation for config values. Add shell metacharacter filtering as defense-in-depth. |

### RC-001-REV-026: Wallet Balance Race Condition via Redis-First Async DB Update

| Field | Detail |
|---|---|
| **ID** | RC-001-REV-026 |
| **Title** | Wallet Balance Race Condition via Redis-First Async DB Update |
| **Severity** | Critical |
| **Bound Function** | Wallet Operations (BF-010), Exchange Flow (CF-010), Wallet Update (SF-033) |
| **Entry Point** | `WalletServiceImpl.update()` called by all financial operations (exchange, withdrawal, C2C, contract) |
| **Sensitive Point / Risk Sink** | `WalletServiceImpl.java` lines 82–87: Redis-first update pattern with async DB sync; `WalletConsumeServer.java` lines 31–78: async consumer with 100ms throttle per partyId |
| **Control Missing / Failure** | No distributed lock, no CAS/optimistic versioning, no SELECT FOR UPDATE, no negative balance guard |
| **Evidence & Data Flow** | Source: Exchange `open` (lines 117–123) calls `walletService.update()` with negative amount. Transit: `WalletServiceImpl.update` (lines 82–87) reads wallet from Redis, adds delta, writes back to Redis, pushes async message. Sink: `redisHandler.setSync()` overwrites previous value; `WALLET_QUEUE_UPDATE` async consumer may reorder or lose messages. Same pattern in `updateExtend()` (lines 154–164). |
| **Why Confirmed** | Classic TOCTOU race condition: two concurrent operations on same partyId can both read same Redis balance, both pass sufficiency check, both apply deltas; second write overwrites first. |
| **Business Impact** | Double-spending: two concurrent exchange orders can both see sufficient balance, both deduct from the stale Redis value, resulting in negative balance or wallet inflation. The async DB consumer receives amount deltas, not source-of-truth values, so final DB balance depends on processing order. This is the most critical financial vulnerability in the platform. |
| **Remediation** | Implement database-level atomic balance check-and-decrement (`UPDATE wallet SET money = money - ? WHERE party_id = ? AND money >= ?`). Add distributed lock per user (Redisson). Add negative balance guard. Replace Redis-first pattern with DB-first with Redis cache invalidation. |

### D. Top Remediation Priorities

1. **Implement atomic balance operations**: Replace Redis-first async pattern with database-level atomic `UPDATE ... WHERE money >= ?` across all wallet operations.
2. **Deploy distributed locking**: Replace all in-memory JVM-local locks (ExchangeLock, C2cLock, ContractLock) with Redis-based distributed locks.
3. **Fix path traversal and file upload**: Implement path canonicalization and re-enable file upload validation controls.
4. **Upgrade authentication stack**: Replace MD5 with bcrypt, migrate token to Authorization header, implement CSRF protection.
5. **Switch to default-deny authorization**: Set `IS_PROTECT_ALL_RESOURCE = true` and audit all endpoint registrations.
6. **Update dependencies**: Prioritize Spring Security, commons-collections, and other CVE-affected libraries.
7. **Implement secrets management**: Move all credentials out of source-controlled configuration files.
8. **Remove conditional 2FA bypass**: Make secondary authentication mandatory for all withdrawal operations.
