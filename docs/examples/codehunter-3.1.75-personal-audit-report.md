## Executive Summary

This report presents the results of a comprehensive security audit of the Anti-Counterfeiting and Traceability System, a PHP 5.x web application supporting product anti-counterfeiting code management, supply-chain traceability, agent/distributor self-service, and administrative operations.

The audit reviewed **69** candidate findings across five structured contract phases. Of these:

- **52** findings are **Confirmed** — backed by direct source code evidence and verifiable control gaps.
- **13** findings are **Highly Likely** — strong circumstantial evidence exists but one or more key confirmatory pieces are missing.
- **4** findings **Need More Evidence** — current evidence is insufficient for confirmation; additional inputs required.
- **0** candidates were **Rejected / Excluded**.

The confirmed findings span **44** generic (CWE-mapped) risk categories and **25** business-logic risk categories. The most critical systemic issues include:

- **Pervasive SQL Injection**: All database queries use raw string concatenation with the deprecated `mysql_*` extension. No parameterized queries or prepared statements exist anywhere in the application. Multiple public-facing and authenticated endpoints are directly exploitable.
- **Obfuscated Core Dependencies**: The file `data/function.php` uses multiple layers of `eval(base64_decode(...))` and `str_rot13` obfuscation, making the full implementation of critical functions (including `get_site_config()`) impossible to audit. This represents a potential hidden backdoor risk.
- **Unauthenticated File Upload**: The KindEditor WYSIWYG upload endpoint (`manage/editor/php/upload_json.php`) lacks authentication checks, allowing unauthenticated arbitrary file upload to the server.
- **No CSRF Protection**: All state-changing operations across admin and agent portals lack CSRF tokens, making every form submission vulnerable to cross-site request forgery.
- **Weak Cryptography**: All passwords use unsalted MD5 hashing. No password policy enforcement exists for minimum complexity or rotation.
- **Missing Audit Trail**: Critical operations including code deletion, export, config changes, and admin account CRUD have no audit logging whatsoever.
- **Business Logic Gaps**: Publicly accessible backend test page bypasses all query controls; agent boxing lacks ownership validation; shipping/return operations lack idempotency; admin account creation allows operator-to-super-admin privilege escalation.

### 1. RC-001-REV-001 — Universal SQL Injection via Raw String Concatenation in All Database Queries

**Severity**: CRITICAL  
**Issue Type**: SQL Injection  
**Bound Function**: Public Anti-Counterfeiting Code Query (`BF-001`)  

**Entry Point**: User-supplied anti-counterfeiting code from POST/GET parameter 'bianhao' is directly interpolated into SQL query without sanitization or parameterization.

**Control Gap**:
- Failure Mode: `completely_missing`
- Expected Controls: input_validation, parameterized_query, output_encoding
- Observed Controls: None
- Why Control Failed: No input sanitization, parameterized queries, or prepared statements detected in the application

**Evidence**:
- `ts/fwm_ts.php (LL93)`: $bianhao = trim($_POST["bianhao"]); - user input received unsanitized
- `ts/fwm_ts.php (LL107)`: $sql="select * from tgs_code where bianhao='$bianhao' limit 1"; - direct interpolation
- `ts/fwm_ts.php (LL482-483)`: $sql = "insert into tgs_history set keyword='".$bianhao."',results='".$results."',..."; mysql_query($sql);
- `manage/index.php (L50)`: $sql="select * from tgs_admin where username='".$username."' and password='".md5($password)."'";
- `ts/fwm_ts.php (L107)`: SQL injection candidate via POST bianhao parameter at public-facing query page

**Confirmation Rationale**: Source code evidence confirms Universal SQL Injection via Raw String Concatenation in All Database Queries. Human-readable source at ts/fwm_ts.php shows direct unsanitized input flow to sink. No controls observed.

**Business Impact / Unauthorized Result**: Exploitation of Universal SQL Injection via Raw String Concatenation in All Database Queries - attacker can execute arbitrary SQL, execute XSS, or access unauthorized data

**Remediation**:
- Implement server-side input validation and sanitization for all user-supplied parameters
- Replace all raw string-concatenated SQL queries with parameterized prepared statements (e.g., PDO or MySQLi with bound parameters)
- Apply context-appropriate output encoding (HTML entity encoding, JavaScript escaping) before rendering user-supplied or database-sourced content in HTML/JS responses

## Appendix

### A. Audit Methodology

The audit was conducted across five structured contract phases:

1. **Project Profile**: System boundary identification, technology stack assessment, dependency inventory, and initial uncertainty enumeration.
2. **Feature Inventory**: Business function decomposition, sub-function mapping, entry point identification, and flow chain construction.
3. **Generic Risk Analysis**: CWE-mapped vulnerability candidate identification with evidence collection and preliminary risk level assignment.
4. **Business Risk Analysis**: Business-logic vulnerability candidate identification focusing on state machines, ownership, idempotency, multi-tenancy, and audit trails.
5. **Finding Review & Confirmation**: Evidence-backed confirmation review with four-tier status assignment (Confirmed, Highly Likely, Needs More Evidence, Rejected/Excluded).

### B. Severity Classification

| Severity | Definition |
| --- | --- |
| Critical | Direct, unauthenticated exploitation leading to full system compromise, arbitrary code execution, or complete data exposure |
| High | Authenticated or unauthenticated exploitation leading to significant data breach, privilege escalation, or business logic bypass |
| Medium | Exploitation requiring specific conditions, leading to limited data exposure, configuration weakness, or defense-in-depth gap |
| Low | Informational finding, defense-in-depth improvement, or theoretical vulnerability with minimal practical exploitability |
