## Executive Summary

This report presents findings from a static security audit of the MicroCommunity Property Management Platform, a multi-module system comprising a Spring Boot backend, uni-app mobile clients, and WeChat mini-program integrations. The platform handles property fee payments, IoT device management (smart meters and charging stations), OA workflows, user authentication, and community data management.

**27 security findings were reviewed** across generic vulnerability patterns and business logic risks. Results are classified by confirmation status:

| Status | Count |
|---|---|
| **Confirmed** | 14 |
| **Highly Likely** | 3 |
| **Needs More Evidence** | 10 |
| **Rejected / Excluded** | 0 |
| **Total Reviewed** | 27 |

**Key risks include:**
- **5 Critical confirmed findings**: SQL injection via dynamic SQL execution, arbitrary Spring bean invocation through unauthenticated smart meter callbacks, and zero-amount payment bypass allowing fee confirmation without actual payment.
- **7 High confirmed findings**: Hardcoded cryptographic secrets, use of deprecated/insecure encryption modes, excessive JWT filter exclusions creating a large unauthenticated attack surface, client-supplied deduction amount manipulation, and multiple unauthenticated charging station callback endpoints.
- **Systemic authentication weakness**: Over 100 URI patterns are excluded from JWT authentication, including payment callbacks, equipment control endpoints, and IoT notification paths with no alternative authentication mechanism.
- **No rate limiting** on any authentication endpoint, enabling distributed brute-force attacks.

## Confirmed Finding Details

### RC-001-REV-001 — SQL Injection via Dynamic SQL Execution (QueryServiceDAOImplMapper)

| Field | Detail |
|---|---|
| **Finding ID** | RC-001-REV-001 |
| **Title** | SQL Injection via Dynamic SQL Execution (QueryServiceDAOImplMapper) |
| **Severity** | Critical |
| **Confidence** | High |
| **Bound Function** | BF-003 Unified API Gateway → SF-004 Dynamic SQL Execution |
| **Entry Point** | `POST /app/{service}` — AppController.servicePost |
| **Sensitive Point / Risk Sink** | `QueryServiceDAOImplMapper.xml:35-37` — `${value}` direct SQL interpolation in `executeSql` and `updateSql` |
| **Control Failure** | Completely missing. No parameterized queries (`#{}`), no SQL whitelist, no input sanitization. The privilege check only verifies URL-based access, not SQL content. |

**Evidence and Data Flow:**

1. **Source:** `AppController.java:82-114` — `@RequestMapping(path = "/{service}", method = RequestMethod.POST)` accepts service code from URL and passes request body to `ApiServiceSMOImpl`.
2. **Transit:** `ApiServiceSMOImpl.java:77-139` — Loads SQL template from `c_service_sql` database table, injects user-supplied parameters, and routes to internal HTTP call.
3. **Transit:** `ApiServiceSMOImpl.java:580-631` — `invokeBusinessSystem` sends HTTP POST to `http://127.0.0.1:8008/cmd/{serviceCode}` with transformed request containing SQL from service definition.
4. **Sink:** `QueryServiceDAOImplMapper.xml:35-37` — `<select id="executeSql" parameterType="String">` uses `${value}` which directly interpolates the entire SQL string without parameterization.

**Why This Is Confirmed:**
The `${value}` interpolation in MyBatis is a well-known SQL injection vector. The SQL from `c_service_sql` table is executed with user-influenced parameters. Any user input reaching the SQL definition parameters can inject arbitrary SQL.

**Business Impact:**
An authenticated attacker with access to any service code can read, modify, or delete any database records. If a service code entry is user-controllable, full database compromise is possible.

**Remediation:**
- Replace `${value}` with parameterized queries using `#{}` wherever possible.
- Implement a SQL whitelist for service definitions stored in `c_service_sql`.
- Add input validation on service parameters before SQL execution.
- Restrict database user privileges to least-privilege principle.

### RC-001-REV-006 — Hardcoded Cryptographic Secrets (AES Key, DES IV, Password Salt)

| **Title** | Hardcoded Cryptographic Secrets (AES Key, DES IV, Password Salt) |
| **Severity** | High |
| **Confidence** | High |
| **Bound Function** | BF-001 User Authentication → SF-001 Credential Management |
| **Entry Point** | Source code constant in `AuthenticationFactory.java` |
| **Sensitive Point** | `AuthenticationFactory.java:54-60` — Three hardcoded constants: `PASSWD_SALT = "hc@java110"`, `AES_KEY = "whoisyourdaddy!!"`, `IV_PARAMETER = "12345678"` |
| **Control Failure** | Completely missing. Keys are compile-time constants with no override mechanism. All deployments share the same cryptographic material. |

**Evidence:**
- `AuthenticationFactory.java:54` — `public final static String PASSWD_SALT = "hc@java110";`

### RC-001-REV-007 — AES/ECB Mode Usage for Encryption

| Field | Detail |
|---|---|
| **Finding ID** | RC-001-REV-007 |
| **Title** | AES/ECB Mode Usage for Encryption |
| **Severity** | High |
| **Confidence** | High |
| **Bound Function** | BF-001 User Authentication → SF-001 Credential Management |
| **Entry Point** | `AuthenticationFactory.java:91` — `Cipher.getInstance("AES/ECB/PKCS5Padding")` |
| **Control Failure** | ECB mode is a fundamentally insecure choice. Identical plaintext blocks always produce identical ciphertext. |

**Evidence:**
- `AuthenticationFactory.java:91` — `Cipher cipher = Cipher.getInstance("AES/ECB/PKCS5Padding");`
- ECB mode does not use an initialization vector, making encryption deterministic. Pattern analysis on encrypted data can reveal repeated values.

**Business Impact:**
Pattern analysis on AES-encrypted data can reveal repeated values. Combined with known plaintext, this weakens confidentiality of all AES-encrypted data.

**Remediation:**
- Migrate to AES/GCM or AES/CBC with random IV.
- Re-encrypt all existing data with the new mode.
- Implement authenticated encryption to provide both confidentiality and integrity.

### RC-001-REV-017 — Zero-Amount Payment Bypass Confirms Fee Without Payment

| Field | Detail |
|---|---|
| **Finding ID** | RC-001-REV-017 |
| **Title** | Zero-Amount Payment Bypass Confirms Fee Without Payment |
| **Severity** | Critical |
| **Confidence** | High |
| **Bound Function** | BF-002 Fee Payment → SF-007 Payment Processing |
| **Entry Point** | `POST /app/toPay` — PaymentController.toPay |
| **Sensitive Point / Risk Sink** | `ToPaySMOImpl.java:113-128` — When `money <= 0`, calls `fee.payFeeConfirm` directly without payment gateway |
| **Control Failure** | No minimum amount validation. No mandatory payment gateway for all amounts. No server-side amount recalculation independent of client params. |

**Evidence and Data Flow:**

1. **Source:** `PaymentController.java:89-101` — `/toPay` endpoint routes to `ToPaySMOImpl`.
2. **Control gap:** `ToPaySMOImpl.java:74-83` — `validate()` only checks field presence (communityId, cycles, receivedAmount, feeId, feeName, appId) but not their values.
3. **Sink:** `ToPaySMOImpl.java:113` — `if (money <= 0) { ... calls fee.payFeeConfirm directly without payment gateway`.

**Why This Is Confirmed:**
The `money` variable is derived from `fee.payFeePre` server response, but the parameters driving that calculation (`communityId`, `cycles`, `feeId`) come from the client request. If those parameters can drive the calculation to zero, fees are marked as paid without payment.

**Business Impact:**
Fees can be confirmed as paid without actual monetary transfer. Direct financial loss for the property management company.

**Remediation:**
- Enforce minimum amount validation (amount must be > 0 for any payment).
- Require payment gateway processing for all amounts, including zero.
- Recalculate fee amount server-side using only server-trusted parameters.

### D. Key Systemic Observations

1. **Authentication Boundary Erosion:** Over 100 URI patterns bypass JWT authentication, creating a broad unauthenticated attack surface that spans payment, IoT, and equipment management endpoints.
2. **Cryptographic Debt:** The entire cryptographic stack uses hardcoded keys, deprecated algorithms (DES), and insecure modes (AES/ECB). A comprehensive cryptographic migration is required.
3. **Input Trust Model:** Multiple payment and data processing paths accept client-controlled parameters (amounts, deduction values, bean names, SQL definitions) without server-side validation or recalculation.
4. **IoT Callback Security Gap:** Smart meter and charging station callbacks have no authentication or signature verification, while some payment callbacks (WeChat Pay, AliPay) do have proper signature verification. The inconsistency suggests security was added selectively.
5. **Missing Observability:** No evidence of rate limiting, comprehensive audit logging, or security monitoring across the platform.
