# Glossary

## Accepted Risk

A human decision that a known risk may remain for a defined reason. It should include owner, rationale, and review or expiration expectations.

## Baseline

The persistent security reference for a Team project. It anchors future iterations, findings, reports, and release decisions.

## Code Source

A local Git, remote Git, patch-only, or PR/MR source connected to a Team project.

## External Report

A scanner report imported into Code Hunter so the findings can be normalized, reviewed, and converted into governed Team findings.

## Finding

A reviewed security issue or candidate issue with evidence, severity, confidence, and remediation direction.

## Fix Package

A scoped repair output generated from selected, reviewed findings.

## Iteration

A Team change cycle. It can include requirements, change sets, PR/MR material, scanner reports, findings, tasks, verification evidence, and release readiness.

## Provider Profile

Configuration for the AI provider and model used by Code Hunter review.

## Release Readiness

A Team decision record that evaluates whether a project or iteration can release based on findings, tasks, CI, verification, accepted risk, policy, and owner approval.

## Remediation Task

Developer-ready security work created from a reviewed finding. A task should include owner, evidence, acceptance criteria, and verification requirements.

## SCA

Software composition analysis. In Code Hunter Team, SCA is connected to policy, exception, remediation, verification, and release gate state.

## VEX

Vulnerability Exploitability eXchange style decision evidence. In Code Hunter, use it only when the team can defend why a vulnerable component is not exploitable or why risk is temporarily accepted.
